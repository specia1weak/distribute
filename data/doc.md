1. 数据有两组，dist是在三台物理机-校园网环境下进行，校园网的延迟可能会随着一天的不同时间产生较大波动。最好的情况下ping各个主机是8ms的延迟，但是也有一些时段会突然飙到100ms以上的延迟，这可能对flink集群和实验的稳定性带来较大冲击。数据受延迟波动影响，实验结果可能无法反映理论情况，但可以作为真实场景的测试。
2. single数据是在单台机器上，通过设定三个Taskmanager进行实验，来模拟三台物理机的三个Taskmanager，由于延迟小于1ms且稳定，他的数据可靠性更好，实验结果更容易得出明确的结论。

## 过程记录文件说明
所有数据在窗口大小2s的情况下运行。
1. late_data_slide 是迟到且被丢弃的数据
2. window_stats_slide 是窗口数据，包括窗口的触发时间，延迟，装了多少数据等内容。因为窗口2s的时间，导致count_expected,loss_network参考意义不大，这点后续会说明
3. 数据命名说明，csv文件名包含了一定的参数设置，其中SZ和TP分别是窗口大小Size和Step的缩写。由于当Size=Step的时候变成滚动窗口。我们进行了多个滚动窗口实验，以及一个滑动窗口实验。P3表示在并行度3个任务上进行。
4. 3Dist说明是在真实三台物理机器上运行。0klag表示容忍延迟为0ms，当然2.6k-lag就是2600ms的延迟了
5. 查看code/pyflink/socket_sliding_window.py，你就知道这些文件的写入时机是什么。dist文件夹下有host1，host2，host3的实验数据，将同名文件的数据合并就是一次实验的最终结果。当然single由于在同一台机器的同一个路径写入，所以一个文件就是最终结果。
## 实验数据
数据由code/deliver/log_generator.py产生，并由log_server.py发送，这意味这每次实验都是相同数据，虽然延迟无法控制，但是数据顺序的预设的乱序已经确定，利于实验分析。
实验所使用的数据生成参数是，这意味这1s内会产生大量数据，在窗口大小2s内很可能收到log_id = 1的正常数据以及log_id=400的延迟数据。
```python
TOTAL_LOGS = 20000
# --- 1. 流量模型配置 ---
BASE_RATE = 100  # 基础速率 (条/秒)
BURST_ENABLE = True  # 是否开启流量突发模式
BURST_PERIOD = 20  # 突发周期(秒)，例如每20秒一波高峰
BURST_AMPLITUDE = 2.0  # 突发倍数 (流量变成基础的几倍)
# --- 2. 网络延迟模型 (长尾分布) ---
# 使用对数正态分布 (Log-Normal) 模拟真实网络延迟
# Mu 和 Sigma 是对数域的参数，不是直接的秒数
DELAY_MU = -1.0  # 调整这个值改变平均延迟
DELAY_SIGMA = 0.8  # 调整这个值改变"长尾"程度 (越大，极慢的数据越多)
MIN_DELAY = 0.05  # 物理最小延迟 (50ms)

# --- 3. 异常模拟 ---
DROP_RATE = 0.02  # 丢包率
```

为什么最后几个数据的延迟总是很高，因为socket没数据发送了
